## JS Chapter 10: Debugging

The order the code executes is a stack: it'll run up until the point it needs to grab information from somewhere else (like a function), at which point it'll wait until that somewhere else completes the task it was waiting for. Each execution scope has a variables object that contains all of the variables, functions, etc., within that scope. This gets created, along with creating the variables, functions, etc. This happens before any code is executed within that same scope. This means that if a function is declared below where it is called, it'll still work because the function was created first before any code was executed. Similarly for variables, it is possible to assign values to the variable before the variable is created, but that's only because behind the scenes that variable was created before any code was executed. Just important to understand this only applies to within the same scope. As far as seeing into other scopes, the child has visibility of the parent's scope (ie: the child can see the variables/functions in the parent's scope), but the parent doesn't have visibility into the child's scope (ie: variables declared inside functions cannot be seen outside of that function).

Error objects are created when an error occurs. There are different types of errors, and the name/description/type/line number/file name attached to the error is helpful for debugging. Just read the error messages. Errors can be handled with `try`/`catch`/`finally` statements, or `throw new Error('[string]')`. For example, the stuff within the try statements will be attempted, if it fails because of an error, the statements in catch will run, and finally no matter what happens the statements in finally will run. On the other hand, throwing a new error object can be placed inside conditional statements for example to throw an error if a condition is met (ie: if something unexpected happened based on a given criteria, throw an error).

For debugging, besides `console.log()`, there's also `console.info()`, `console.warn()`, and `console.error()`, which are basically the same but with different formatting as far as I can tell. Console logs can be grouped together into a collapsable list with `console.group()` andend the group with `console.groupEnd()`. `Console.table()` is used to show a table of the data in the object/array. Breakpoints can be set by clicking (or right clicking for conditional breakpoints) on the numbered row labels on the side. Or, `debugger` can be typed to serve the same purpose (and wrapped in a conditional for a conditinal debugger). Hover over variables to see what values are contained in it. The book mentions common errors: tldr is that there's a lot of silly mistakes that can happen, so stay calm, think things through, and try to box the problem into progressively smaller regions until the source is isolated.
