I feel like there's enough form options that I'd be best off looking up how to use each option when I need to use a particular form control. The options/form controls available are single line text box, multiline textbox, radio buttons, checkboxes, file input, password input, dropdown list, and multiple selectable items list. There's some variants on some of these, like for example a text box uses an `<input>` tag (no closing tag) with `type="text"` and name `name=[name]"`, where type is the type of field it is and name is the key in the key/value pair sent to the thing that processes the form; and similarly, an email is also an input, but the type is set to email and name set to email (or whatever the key should be); and similarly for url's, the type would be a url. I wonder if there's some flexibility to this though; like I'm guessing if whatever processes the form expects a different type/name, then maybe that is the true truth of what the fields should be set as. Not sure. The book mentions using label tags to either group the label with the form item (by wrapping the text next to the form item with label tags) or by using the `for` attribute in label to be equal to the id attribute's value of the form item that label's text belongs to. I don't think there's a visual difference, but for screen readers it's helpful. Actually there is one more useful difference; the selectable area will expand if the label wraps the label and radio/checkbox buttons. `<form>` tags surround the form controls with an action attribute pointing to where the form's data gets sent to (all the examples so far are url's ending in php). There's two methods the data can be sent as, as determined by the method attribute; get (for shorter, text-based things, or reading data from a server) or post (for longer text data, files, or modifies database info). Hidden form items are useful for sending a key/value pair with default values hidden from the user. Form items can be grouped together using `<fieldset>` and that grouping labeled with `<legend>`. The required attributed can be used to force a required field to have a value. The submit button is used to submit form data (input field with type attribute set to submit, although setting type to image and adding an src for which image to use might also work).

## HTML Chapter 14: Lists, Tables, and Forms

This chapter introduces a few more css properties. `list-style-type` can be used for changing how the bullets (disc, circle, or square) and numbering (decimal, lower-alpha-, upper-alpha, lower-roman, etc.,) are. `list-style-image` can replace the bullets with an image (lined using url() for the value). `list-style-position` is used to place the bullet within the text (inside) or off to the side causing the block of text to be indented (outside). `list-style` is shorthand for the above properties; similar to how border is shorthand for specific borderp properties. Table properties exist but I don't think they're necessarily table specific; things like width, padding, border, background-color. One interesting thing is css has an even/odd selector, so for example tr.even or tr.odd can be used to format those selected table rows differently. The spacing between cells can have a gap between them, like a margin, except it's called `border-spacing`. `Border-collapse` is used to cause the cell borders to overlap rather than sit side-by-side. `empty-cells` can be used to hide empty cells (set to hide, show, or inherit).

There's recommendations on how to style forms. Lining up the form controls with left justification is usually good. Similarly, the labels are good to line up with left justification in a seperate column to the left, though sometimes right justification works better. Radio and checkboxes should have the label on the right side though.

## JS Chapter 6: Events

Interesting chapter. Events can trigger functions to be called. There's a lot of different events that can occur, like page load ('load'), a key is pressed ('keydown'), the mouse is clicked ('click'), the user selects something ('select'), submits a form ('submit'), etc. How these events call the function can be done in three ways. The first way is to place in the html tag an event attribute (such as `onclick="functionName()"`). This way is not preferred because of a preference to sepearate HTML from javascript. The second way is to use 'traditional dom event handlers', which is `DOMelement.onevent = [functionName]`, but is unable to call multiple functions from a single event. The third way, most preferred, is `DOMelement.addEventListener('[eventName]', [functionName])`. FunctionName has to be a function, without being invoked. This means that to pass parameters to the function, an anonymous function has to be used to call the intended function: ie, in place of `[functionName]`, instead do `function() {[functionName(parametersToPassIn)]}`. Note that the event is automatically passed in, so if I want to use that automatically passed in object in the function that the anonymous function calls, I have to pass it in: `function(event) {functionName(event)}`. Also, the event object contains information such as what element was selected, which is useful for extracting data from a form; with the target info, can read the form using the name fields (ie: `[eventObject].target.[name].value` to grab the value on the element with a particular name.
